PLATEFORME D’ÉCHECS DISTRIBUÉE : RABBITMQ

DESCRIPTION
Ce projet implémente une plateforme d’échecs distribuée basée sur une
architecture événementielle utilisant RabbitMQ.
Le déroulement d’une partie d’échecs est modélisé comme une suite
d’événements envoyés de manière asynchrone et traités par plusieurs
services indépendants.

OBJECTIFS
- Communication asynchrone
- Découplage producteur / consommateurs
- Tolérance aux pannes
- Reconfiguration dynamique
- Architecture orientée événements

ARCHITECTURE
Le système est composé de :
- Un producteur gérant la partie d’échecs
- RabbitMQ jouant le rôle de middleware de messagerie
- Plusieurs services consommateurs indépendants :
  * Validation des coups
  * Analyse de position
  * Enregistrement de la partie
  * Service spectateur (optionnel)

Chaque service possède sa propre queue RabbitMQ.

PRÉREQUIS
- Python 3.9 ou plus
- Docker et Docker Desktop
- RabbitMQ (déployé via Docker)
- Bibliothèque Python : pika

LANCER RABBITMQ
Commande Docker :
docker run -d --name rabbitmq -p 5672:5672 -p 15672:15672 rabbitmq:3-management

Interface web :
http://localhost:15672
Identifiants :
login : guest
password : guest

TOPOLOGIE RABBITMQ
- Exchange : chess.events (type fanout)
- Queues :
  - validation.queue
  - analysis.queue
  - storage.queue
  - spectator.queue

LANCEMENT DU PROJET
1. Lancer les services consommateurs (dans des terminaux séparés) :
   python consumers/validation_service.py
   python consumers/analysis_service.py
   python consumers/storage_service.py
   python consumers/spectator_service.py

2. Lancer le producteur :
   python producer/producer.py

COMPORTEMENT ATTENDU
- Les messages sont stockés si aucun consommateur n’est actif
- Les consommateurs peuvent être lancés après coup
- Les messages non acquittés sont retraités après une panne
- Les services peuvent être ajoutés ou supprimés sans modifier le producteur
